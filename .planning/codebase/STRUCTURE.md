## Directory Purposes

**src/:**
- Purpose: Main library implementation
- Contains: TypeScript source files (all .ts, no .js)
- Key files: `database/index.ts` and `model/index.ts` are 80% of logic
- Build output: Compiles to `lib/` directory

**src/database/:**
- Purpose: Database connection lifecycle and model registry
- Contains: Single `index.ts` file with Database class
- Key responsibility: MongoClient management, model registration, collection setup
- Exports: `Database` class and `ObjectID` type alias

**src/model/:**
- Purpose: Collection abstraction layer
- Contains: Single `index.ts` file with Model class
- Key responsibility: CRUD operations (insert, find, update, delete), aggregation, bulk operations
- Exports: `Model` generic class with 12 public methods

**src/types/:**
- Purpose: TypeScript type definitions and interfaces
- Contains: Three files (index.ts barrel, database.ts, model.ts)
- Key exports: DatabaseConfig, CreateModelProps, ModelValidationSchema, CreateIndexProps
- Pattern: All types named with suffix Props or Schema or Config for clarity

**src/utils/:**
- Purpose: Enums, helpers, and constants
- Contains: Three files (index.ts barrel, enums.ts, database.ts)
- Key exports: METHODS enum (12 operations), CUSTOM_VALIDATION enum, toObjectId() function
- Pattern: Enums exported from enums.ts, helpers exported from domain-named files

**src/schema/:**
- Purpose: Documentation and decorator examples (not actual implementation)
- Contains: Commented example showing intended Schema decorator syntax
- Status: Currently just documentation; @Schema, @Description, @Pre, @Pattern decorators are NOT implemented
- Note: This is a design placeholder, not functional code

**examples/:**
- Purpose: Usage demonstrations for end users
- Contains: Three files showing database connection, model creation, and typical operations
- Connection example: Shows env var overrides and database configuration
- Model example: Shows schema definition, indexes, and pre-method hooks
- Usage example: Shows typical CRUD flow (insert, find, update, delete, findMany, total)
- Status: Runnable with `npm run example`

**lib/:**
- Purpose: Compiled JavaScript and type declarations (generated)
- Generated by: `npm run build` (runs `tsc` + `tsc-alias`)
- Contents: .js files, .d.ts files, .js.map sourcemaps
- Not committed: Listed in .gitignore
- Entry point for npm: `lib/index.js` (from package.json `main` field)

## Naming Conventions

**Files:**
- Database: `src/database/index.ts` (single file per domain)
- Model: `src/model/index.ts` (single file per domain)
- Types: `src/types/<domain>.ts` (one file per conceptual domain: database.ts, model.ts)
- Utilities: `src/utils/<domain>.ts` (organized by responsibility: enums.ts, database.ts)
- Indexes are barrel exports: `src/types/index.ts`, `src/utils/index.ts` (re-export all)
- Examples: `examples/<domain>/<file>.ts` (with domain subdirectories for models)

**Functions:**
- Camel case: `toObjectId()`, `setupCollections()`, `findOneAndDelete()`
- Pre-method naming: `preMethod[METHODS.INSERT]` (enum-based lookup)
- Private method naming: Prefixed with underscore or Symbol-keyed: `_update`, `[kGetDbName]()`
- Async functions: Standard async/await, no suffix; callers use `await`

**Variables:**
- Camel case for public: `collectionName`, `documentDefaults`, `allowedMethods`
- Private symbols: `kClient`, `kDb`, `kDatabase`, `KModelMap` (follow Symbol convention in TypeScript)
- Symbol-keyed private fields use const before Symbol call: `const kClient = Symbol('kClient')`

**Types:**
- Interfaces with suffix: `DatabaseConfig`, `CreateModelProps`, `ModelValidationSchema`
- Enums with PascalCase: `METHODS`, `CUSTOM_VALIDATION`
- Generics with uppercase: `Model<ModelType extends Document>`
- Type aliases lowercase when simple: `ObjectID = ObjectId`

## Where to Add New Code

**New Feature (e.g., new CRUD operation):**
- **Primary code:** Add method to `src/model/index.ts` (e.g., `async findByIdAndDelete()`)
- **Types:** Add parameter types to `src/types/model.ts` if needed
- **Enum:** Add operation to `METHODS` enum in `src/utils/enums.ts` (line 1-14)
- **Pre-hooks:** Automatically available in `preMethod` Record if new operation follows pattern
- **Export:** Re-export from `src/index.ts` (already exports Model)
- **Example:** Add usage to `examples/model/usage.ts`

**New Database Feature (e.g., session management, bulk operations):**
- **Primary code:** Add method to `src/database/index.ts` (e.g., `getSession()`)
- **Export:** Re-export from `src/index.ts`
- **Types:** Add to `src/types/database.ts` if creating new config option
- **Example:** Add to `examples/connection.ts`

**New Type/Interface:**
- **Location:** `src/types/<domain>.ts` (or expand existing domain file)
- **Re-export:** Add to `src/types/index.ts` barrel export
- **Pattern:** Name with descriptive suffix (Config, Props, Schema)

**New Utility/Helper:**
- **Location:** `src/utils/<domain>.ts` (group with related utilities)
- **Re-export:** Add to `src/utils/index.ts`
- **Pattern:** Functions with `export function`, enums with `export enum`

**New Validation Rule:**
- **Location:** Add case to `CUSTOM_VALIDATION` enum in `src/utils/enums.ts`
- **Implementation:** Use in schema definitions when creating models
- **Pattern:** Client-side constraint definition; MongoDB enforces via JSON Schema

**Breaking Changes (rare):**
- **Coordinate:** These affect npm package semver
- **Location:** All changes are in `src/`; update version in `package.json` (line 3)
- **Example:** Changing Model constructor signature would require major version bump

## Import Path Aliases

**Configured in tsconfig.json (lines 42-48):**

| Alias | Maps to | Use case |
|-------|---------|----------|
| `@/*` | `src/*` | Absolute imports for library code |
| `@examples/*` | `examples/*` | Examples can use absolute paths |
| `@test/*` | `test/*` | Tests (if added) |
| `@types/*` | `src/types/*` | Shorthand for type imports |
| `@utils/*` | `src/utils/*` | Shorthand for utility imports |

**Current usage in codebase:**
- Model imports Database: `import { Model } from '@/model'`
- Database imports types: `import { DatabaseConfig, ModelSetup } from '@/types'`
- Database imports utils: `import { METHODS } from '@/utils/enums'`
- Model imports utils: `import { toObjectId } from '@/utils'`
