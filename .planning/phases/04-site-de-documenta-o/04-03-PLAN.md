    - `sanitizeFilter(filter)` é OPT-IN (SEC-01/D-06): nenhum método do Model o chama sozinho; o dev o aplica a input não-confiável (ex.: `req.query`) antes de passar a `findMany`/`find`. Mostrar o exemplo `const safe = sanitizeFilter(req.query); await User.findMany(safe);`.

    - Contraste: o guard `$where` é INCONDICIONAL e NÃO-desligável (D-05) — qualquer filtro contendo `$where` (em qualquer profundidade) é rejeitado com `MongoatValidationError` (`code: FORBIDDEN_OPERATOR`), independente de `sanitizeFilter`. Cross-link para handle-errors.

    `handle-errors.md` — "Handle errors":
    - Tabela subclasse → `.code` → quando é lançada (de MIGRATION.md §2.1): `MongoatValidationError` (`INVALID_OBJECT_ID`, `FORBIDDEN_OPERATOR`), `MongoatConnectionError` (`NOT_CONNECTED`, `MISSING_DB_NAME`), `MongoatDriverError` (`DUPLICATE_KEY`, `DRIVER_ERROR`).

    - `.cause` preserva o erro original do driver; mensagens são sanitizadas (sem stack/valor duplicado — SEC-03). Nota: métodos de leitura REJEITAM a Promise quando desconectados (não lançam síncrono) — usar try/catch em torno do `await`.
