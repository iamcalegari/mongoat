**Como** mantenedor do Mongoat rumo à v1.0, **quero** um core testado de ponta a ponta, verificado por CI e blindado contra injeção e entrada insegura, **para que** a lib publicada seja um portão de qualidade confiável para o dev que a consome.

    - src/model/hooks.ts §runPostHooks — o ramo `fireAndForget` faz `Promise.resolve().then(fn).then(...).catch((err) => onHookError(err, ctx))`. Se o `onHookError` do dev lançar (síncrono) ou retornar Promise rejeitada, vira unhandledRejection.
    - test/model/hooks-fire-and-forget.test.ts (padrão de teste de fireAndForget + onHookError)

  </read_first>
  <files>src/model/hooks.ts, test/model/hooks-onhookerror-throws.test.ts</files>
  <action>
    Em `runPostHooks`, blindar o `.catch((err) => onHookError(err, ctx))` do ramo `fireAndForget`: envolver a invocação de `onHookError` de modo que qualquer erro que ELE próprio lance (síncrono) OU rejeite (assíncrono) seja contido e nunca escape como `unhandledRejection`. Ou seja, o handler passa a ser algo como `.catch((err) => { try { const r = onHookError(err, ctx); if (r && typeof (r as { then?: unknown }).then === 'function') (r as Promise<unknown>).catch(() => {}); } catch { /* último recurso: onHookError falhou — não há para onde propagar sem quebrar o caller */ } })`. Manter o comportamento existente quando `onHookError` NÃO lança (a rejeição original ainda é roteada a ele). Não alterar o caminho normal (não-fireAndForget), que deve continuar propagando (D-05).

    Criar test/model/hooks-onhookerror-throws.test.ts: registrar um post-hook `fireAndForget` que rejeita, com um `onHookError` que ELE MESMO lança; asserir que nenhum `unhandledRejection` é emitido (capturar via `process.on('unhandledRejection')` durante o teste, com um pequeno flush de microtasks/timer) e que o `caller` da operação CRUD resolve normalmente.
  </action>
