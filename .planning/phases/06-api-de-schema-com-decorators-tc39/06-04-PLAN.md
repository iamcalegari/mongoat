## Phase Goal

**As a** desenvolvedor que consome a lib, **I want to** declarar hooks direto na classe de schema via `@Pre`/`@Post` (nível de classe) e `@Pre` de campo (transformação do valor, ex.: hashPassword), **so that** eu registro comportamento de transformação/validação junto da declaração do schema, com ordem de execução determinística e reaproveitando o pipeline de hooks já existente.

> Fatia vertical (Wave 3): a última capacidade da API de decorators — hooks declarativos na classe, fechando DECO-02.

<objective>
Adicionar os decorators de hook — `@Pre` (classe e campo, D-09), `@Post` (classe, D-10) — que apenas REGISTRAM hooks no pipeline existente da Fase 2 (nenhum novo dispatch), com ordem determinística D-11 e política de erro híbrida D-14 (erro local de método inexistente estoura na decoração). Ao fim, um dev declara `@Pre('insert', hashPassword)` num campo e o hook dispara na ordem correta contra Mongo real.

Purpose: Fecha DECO-02 e completa a paridade da API de decorators com a API de objetos (que já tem `hooks`/`.pre()`/`.post()` desde a Fase 2). Sem isto, a classe decorada só declara shape, não comportamento.
Output: `@Pre`/`@Post` em `src/schema/decorators.ts`, `extractDecoratorHooks` em `compile.ts`, wiring de ordem no constructor do Model, guard `INVALID_HOOK_METHOD`, exports e testes de ordem/erro.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| declaração de hook na classe → pipeline de hooks → driver | Um hook de segurança (hash/auditoria) que não dispara, ou dispara fora de ordem, quebra uma garantia de segurança/integridade dos dados |
| nome de método de hook (string) → registro no pipeline | Um método inexistente registraria um hook que nunca dispara, silenciosamente |
