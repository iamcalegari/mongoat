## Phase Goal

**As a** desenvolvedor que consome a lib, **I want to** usar um `@Pre` de campo com transform assíncrono (ex.: `hashPassword`) e compilar classes decoradas aninhadas totalmente opcionais, **so that** o valor transformado é persistido corretamente (nunca uma Promise) e o schema aninhado é aceito pelo MongoDB — fechando os 2 gaps de verificação da Fase 6.

> Fatia de fechamento de gaps (Wave 1): repara as capacidades DECO-02 (`@Pre` de campo assíncrono) e DECO-03 (equivalência de classe aninhada totalmente opcional) já entregues em 06-01..06-04, sem renumerar nem alterar aqueles planos. Cada tarefa entrega correção + teste de regressão + prova de integração — uma fatia vertical completa.
>
> Nota de processo: a linha `**Goal**` da Fase 6 no ROADMAP não está no formato User Story (flag registrada em 06-VERIFICATION.md `mvp_mode_note`); esta história é escopada às capacidades reparadas, seguindo o precedente dos planos 06-02..06-04 desta mesma fase. Reconciliar retroativamente com `/gsd mvp-phase 6` é opcional e fora do escopo desta correção.

<objective>
Fechar os 2 gaps de 06-VERIFICATION.md, ambos localizados em `src/schema/compile.ts`:

1. **Gap 1 (CR-01 + WR-05):** o wrapper do `@Pre` de campo em `extractDecoratorHooks` é síncrono e grava `document[field] = fn(document[field], ctx)` sem aguardar — quando `fn` é `async` (o exemplo canônico `hashPassword` do JSDoc de `@Pre`), o campo fica com uma Promise pendente que o BSON serializa errado (objeto vazio/descartado). Além disso, o wrapper materializa `document[field]` mesmo quando o campo está AUSENTE, mascarando a validação `required`.
2. **Gap 2 (WR-06):** `compile()` sempre emite `required`, inclusive quando vazio. No nível raiz isso é mascarado (o `schemaValidatorBuilder` anexa `_id`), mas uma classe aninhada totalmente opcional (via `@Prop({ type })` ou `items:`) embute `required: []`, que o `$jsonSchema` do MongoDB REJEITA no `createCollection`/`collMod` — quebrando a equivalência DECO-03 nesse shape.

Purpose: Promover DECO-02 e DECO-03 de "parcialmente satisfeito" a "satisfeito" — o `@Pre` de campo (exemplo-bandeira: hashing de senha) para de corromper o dado, e a API de decorators volta a ser intercambiável com a API de objetos até no caso extremo aninhado-opcional.
Output: Correções cirúrgicas em `extractDecoratorHooks` e `compile()` (`src/schema/compile.ts`); teste de regressão de `@Pre` de campo async + campo ausente; casos aninhados totalmente opcionais em `nested-compile.test.ts`; teste de integração de `setupCollection` contra Mongo real.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| transform de `@Pre` de campo (hook de segurança, ex.: hashPassword) → document → BSON/MongoDB | Um transform que deixa uma Promise pendente ou é pulado silenciosamente corrompe/descarta o campo protegido (senha em claro ou perdida) |
| campo `required` com `@Pre` de campo → validação `$jsonSchema` do MongoDB | Materializar um campo ausente via `fn(undefined)` faz o `required` passar para um campo que deveria faltar — invariante de integridade quebrada |
| classe decorada aninhada totalmente opcional → validator `$jsonSchema` → `createCollection`/`collMod` | Um `required: []` aninhado inválido rejeita o setup da coleção, quebrando a equivalência das duas APIs (correção/disponibilidade) |
