## Phase Goal

**As a** desenvolvedor que consome a lib, **I want to** definir schemas de model com decorators TC39 padrão (`@Schema`/`@Prop`/açúcares) como alternativa de primeira classe à API de objetos, **so that** eu ganho a ergonomia de schema-como-classe compilando exatamente para o mesmo `ModelValidationSchema` interno, sem `reflect-metadata` nem flags experimentais no tsconfig.

> Derivada da linha `**Goal**` da ROADMAP.md (Fase 6). Esta é a fatia-esqueleto (Wave 1): a menor fatia vertical que prova a pilha inteira — build, runtime em node real e compile — funciona ponta-a-ponta.

<objective>
Estabelecer o módulo `src/schema/` com os decorators canônicos mínimos (`@Schema`, `@Prop`) e `Schema.compile`, provando que a cadeia de build de produção (tsdown/Rolldown/Oxc + `@rolldown/plugin-babel`) faz o lowering da sintaxe de decorators stage-3 e que uma classe decorada executa em **node real** — o maior risco técnico da fase (RESEARCH.md, Pitfall 1). Ao fim, um dev escreve uma classe decorada e chama `Schema.compile(Classe)` obtendo o mesmo `ModelValidationSchema` do objeto plano equivalente.

Purpose: Sem o build lowering validado empiricamente ANTES de qualquer outra task, todo o resto da fase pode ser invalidado tarde (o pacote publicado quebraria com SyntaxError em node). Esta fatia gateia o build e entrega a capacidade pública `Schema.compile` (D-07).
Output: módulo `src/schema/` (polyfill, guards, decorators @Schema/@Prop, compile), tipo `SchemaClass<T>`, config de build com babel, exports no barrel, script de smoke de build e testes de equivalência/guard.
</objective>
